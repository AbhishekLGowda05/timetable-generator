# Frontend roadmap — React + TypeScript (ERP style)

Nice — I’ll give a *practical, production-minded, step-by-step roadmap* you can follow to build the timetable frontend using the same stack/patterns used by your ERP. I’ll assume React + TypeScript + Vite + Tailwind + Zustand (the stack in your admin README). For exact local setup commands and environment details refer to your ERP frontend README. 

---

## High-level plan (phases)

1. **Bootstrap & env** — create project, match ERP conventions, make it relocatable.
2. **Core layout & routing** — global shell, auth / admin guard, top nav + sidebar.
3. **Data & API plumbing** — API client, schemas, services, state stores.
4. **Upload + Parse UI** — upload CSV/XLSX, preview, parse to JSON.
5. **Constraints UI** — grid timings page, global constraints page, class-level constraints.
6. **Generate / Job flow UI** — submit job, status, live logs, diagnostics + results view.
7. **Timetable viewer** — class / teacher / room views, exports, manual edit mode.
8. **Quality, tests & performance** — unit tests, E2E, perf tuning.
9. **Bundle & deploy** — build scripts, env, docs for relocation & integration.

# Phase 1 — Bootstrap & environment (relocatable)

Goal: a dev-ready frontend folder that can be dropped into any repo.

1. existing folder `frontend/` (so it’s movable) and initialize with the same conventions as your ERP:

   * `pnpm init vite` (React + TS template)
   * `pnpm add -D typescript tailwindcss postcss autoprefixer`
   * `pnpm install` then `npx tailwindcss init -p`
2. Copy the ERP `tsconfig.json`, ESLint, Prettier, and Vite config (so style + lint rules match). This keeps the code relocatable.
3. Add `README.local.md` describing:

   * `pnpm install`
   * `pnpm run dev` (default port and how to change)
   * required `.env` keys (see next section)
4. Create `.env.example` with:

   ```
   VITE_API_URL=http://localhost:8010    # solver backend
   VITE_ADK_API_URL=http://localhost:8004 # optional agent/ADK
   ```
5. Create a small `relocate.md` describing how to copy this folder into the main monorepo (what to update: package scripts, base path env).

**Output:** A movable `timetable-frontend` directory that follows ERP dev conventions.

# Phase 3 — Data schemas, types & API plumbing

Goal: a single source of truth for data shapes and HTTP calls.

1. **Types & schemas**

   * `src/schemas/` with TypeScript interfaces that mirror backend JSON schema:

     * `School`, `ClassSection`, `Subject`, `Teacher`, `Room`, `ConstraintConfig`, `SolverJob`, `SolverResult`.
   * Keep these aligned with the server `shared/schemas` files (you already have similar on server); small script later can sync them.
2. **API client**

   * `src/services/api.ts` — Axios instance (base URL from `VITE_API_URL`).
   * `src/services/timetableApi.ts` — functions:

     * `uploadRawData(file) -> { uploadId }`
     * `validatePayload(payload) -> { valid: boolean, errors }`
     * `createJob(payload) -> jobId`
     * `getJobStatus(jobId) -> status`
     * `getJobResult(jobId) -> result`
     * `cancelJob(jobId)` etc.
3. **State management**

   * Use the same pattern as ERP — **Zustand** stores:

     * `useUploadStore` (upload progress, parsed preview)
     * `useConstraintsStore` (current toggles + settings)
     * `useJobStore` (current jobId, status, logs, result)
   * Persist constraints in localStorage (so user doesn’t re-enter while building).
4. **Schema validation on front-end**

   * Use `zod` to validate the payload before hitting the backend validator — faster UX and better error messages.

**Deliverable:** Strong typed API layer and stores so pages can call services deterministically.

# Phase 4 — Upload + parse UX

Goal: upload CSV/XLSX, preview parsed JSON, allow mapping adjustments.

1. **FileUploader** component:

   * Accept `.csv` and `.xlsx`.
   * Show template sample download (CSV templates for each entity).
   * Show progress bar; allow cancel.
2. **Client-side parse preview**

   * Parse small files on client using `papaparse` (CSV) and `xlsx` (XLSX).
   * Show interactive preview table (first 20 rows) for each sheet/entity.
   * Allow user to map column names if they use slightly different headers.
3. **Send to backend**

   * Convert parsed data to the canonical JSON, call `validate` endpoint first.
   * If server validation fails, show human-friendly errors with pointer to row/column.
4. **UX niceties**

   * Show sample fabricated datasets link (you said you’d generate them).
   * “Save as default mapping” for repeated uploads.

**Deliverable:** Upload flow that yields a validated JSON payload ready for job submission.

# Phase 5 — Constraints UI (the core)

Goal: UI that maps exactly to solver constraints and remains user-friendly.

1. **Three stages UI** (wizard or 3 cards):

   * **Grid Timing**: school hours, period lengths, weekdays, Saturday count, recess windows.
   * **Global constraints**: toggles for language-blocks, max consecutive, core before lunch, resource counts, block-period flags.
   * **Teacher/Class constraints**: per-teacher availability matrix (day × slot), per-class subject mapping, resource assignments.
2. **Component details**

   * `ConstraintToggleRow` with short description + “What this means” expanded example.
   * `TeacherAvailabilityMatrix` — clickable grid like Excel; copy/paste availability, bulk actions.
   * `ResourceConfig` — for labs, optional instance counts, block-length selectors.
   * `LanguageBlockEditor` — quick assign for language teachers per section (auto-suggest from data).
3. **Validation**

   * Real-time front-end validation using zod: e.g., if language block ON → ensure each section has at least N language teachers assigned.
4. **Save / Load**

   * Save constraint presets (name them), so admins reuse common configurations.
   * Export/Import constraint JSON.

**Deliverable:** An admin-friendly interface where every solver option is available and explained.

# Phase 6 — Generate / Job orchestration UI

Goal: submit the job, show real-time status/logs, allow async operation.

1. **Submit flow**

   * `Generate` button bundles: parsed data + constraints → POST `/api/jobs/create`.
   * Receive `jobId`; show start timestamp and estimated ETA (based on historical runtime).
2. **Job status panel**

   * Poll `GET /api/jobs/:id/status` every 2s initially then backoff.
   * Show statuses: `queued`, `running`, `failed`, `partial`, `completed`.
   * Stream logs if available: `GET /api/jobs/:id/logs` (paginated).
3. **Diagnostics**

   * If `failed`, fetch `/api/jobs/:id/diagnostics` which maps back to constraint violations; show actionable suggestions (e.g., “No teacher available for Lab on Monday → relax block or add resource”).
4. **Concurrent jobs**

   * If user starts multiple jobs, show queue list with ability to cancel or prioritize.
5. **Notifications**

   * Desktop toast + email (if configured) on completion/failure.

**Deliverable:** A robust async job flow UI — users can submit and continue working while solver runs.

# Phase 7 — Timetable viewer + editor

Goal: show generated timetable in class/teacher/room views; allow manual edits that validate against constraints.

1. **Data model**

   * `WeekGrid`: days × periods matrix for a section.
   * Precompute teacher and room cross-views on the server OR derive client-side from master JSON.
2. **Views**

   * `ClassView` — grid UI with drag-and-drop swapping enabled.
   * `TeacherView` — list of teachers → click → highlight their weekly blocks.
   * `RoomView` — show resource bookings.
3. **Manual edit mode**

   * When admin drags a subject to a slot:

     * Call `POST /api/validate-change` with the proposed change.
     * Server responds: `{ ok: true }` or `{ ok: false, reason }`.
     * If ok → commit the change locally and optionally persist via `PUT /api/jobs/:id/commit-change`.
   * Provide “Suggest best swap” using server endpoint to auto-fix small conflicts.
4. **Export/print**

   * Export JSON, CSV, printable PDF, and `.ics` per teacher/class.
   * Add “Publish” toggle (published timetables become read-only for students/teachers).

**Deliverable:** A full featured timetable renderer and admin editor with safe validation.
